title: "Metadata"

---

Use `build.yml` to describe what your repository should build. The generator reads this file from the repository root.

## Schema validation

Add this comment to the top of your `build.yml` for editor validation:

```yaml
# yaml-language-server: $schema=https://raw.githubusercontent.com/eddietisma/buildcharts/main/schemas/v1beta.json
```

## Top-level fields

### `version`

String value for the config version. Use `v1beta`.

### `variables`

Variables become Bake variables and are also exposed through the generated `_common` target args.

<Tabs>
  <Tab title="Scalar list">
    ```yaml
    variables:
      - VERSION
      - COMMIT
    ```
  </Tab>
  <Tab title="Inline defaults">
    ```yaml
    variables:
      - VERSION: "1.0.0-local"
      - COMMIT: ""
    ```
  </Tab>
  <Tab title="Expanded defaults">
    ```yaml
    variables:
      - VERSION:
          default: "1.0.0-local"
      - COMMIT: ""
    ```
  </Tab>
  <Tab title="Mapping">
    ```yaml
    variables:
      VERSION: "1.0.0-local"
      COMMIT: ""
    ```
  </Tab>
</Tabs>

### `plugins`

Optional list of plugin names to run during generation:

```yaml
plugins:
  - NuGetAuthenticate@v1
  - TestcontainersDinD@v1
```

See [Plugins](/core-features/plugins) for details.

### `targets`

Required mapping of source path to one or more target definitions.

You must define exactly one `build` target across the whole file.

<AccordionGroup>
  <Accordion title="Single scalar">
    The simplest form — map a source path directly to a type name:

    ```yaml
    targets:
      buildcharts.sln: build
    ```
  </Accordion>
  <Accordion title="Mapping with one type">
    Use a mapping when you need to set `with` keys:

    ```yaml
    targets:
      src/App/App.csproj:
        type: docker
        with:
          base: mcr.microsoft.com/dotnet/runtime:10.0
          tags: ["docker.io/example/app:${VERSION}-${COMMIT}"]
    ```
  </Accordion>
  <Accordion title="Mapping with multiple types">
    Assign multiple types that share the same `with` block:

    ```yaml
    targets:
      src/App/App.csproj:
        type: [build, test]
        with:
          base: mcr.microsoft.com/dotnet/sdk:10.0
    ```
  </Accordion>
  <Accordion title="Sequence of entries">
    Use a sequence when each type needs its own `with` block:

    ```yaml
    targets:
      src/App/App.csproj:
        - type: build
          with:
            base: mcr.microsoft.com/dotnet/sdk:10.0
        - type: docker
          with:
            base: mcr.microsoft.com/dotnet/runtime:10.0
            tags: ["docker.io/example/app:${VERSION}-${COMMIT}"]
    ```
  </Accordion>
</AccordionGroup>

### `types`

Optional type-level matrix expansion. Use `types.<type>.matrix` when you want to expand every target of a given type across one or more axes:

```yaml
types:
  docker:
    matrix:
      platform: ["linux/amd64", "linux/arm64"]
```

Every target of that type is expanded across the matrix axes. Each axis value is exposed to the chart as an uppercase snake case build arg. In this example, the generated Bake plan exposes `PLATFORM`.

## `with` keys

The schema allows free-form objects under `with`, but the current generator emits these keys into the Bake plan:

| Key | Effect |
|---|---|
| `base` | Adds a named `base` context using `docker-image://...` |
| `tags` | Sets Bake tags for the generated target |
| `dockerfile` | Overrides the default chart Dockerfile path |
| `allow` | Sets BuildKit entitlements such as `network.host` |
| `args` | Adds per-target build args |

<Note>
  `args` values can be strings or arrays. Array values are emitted as comma-separated strings in the generated HCL.

  If you omit `dockerfile`, BuildCharts uses `./.buildcharts/<chart-name>/Dockerfile`.
</Note>

## Alias resolution

Each target `type` must match an alias in `charts/buildcharts/Chart.yaml`. If you define `type: publish`, you also need a chart dependency whose `alias` is `publish`.
